# DecisionLab 项目整体实施方案

## 1. 项目概述

### 1.1 项目名称
DecisionLab 需求决策智能体平台

### 1.2 项目定位
本项目面向需求研究、产品分析和质量改进场景，建设一套以智能体和流程编排为核心的决策平台。系统围绕“需求澄清、问卷生成、数据采集、决策分析、报告输出”形成闭环，支持研究人员以较低技术门槛完成结构化决策工作。

### 1.3 与交付清单的对应关系
结合本项目建设范围，本项目可映射到交付清单中的三类核心软件能力：

1. 智能问卷生成模块：已实现，以 `KANO_QFD` 场景最贴合当前主链路。
2. 客户反馈数据收集模块：已实现，支持公开问卷、匿名提交、版本化采集与统计展示。
3. 可视化报告生成模块：已实现，支持管线报告与项目总报生成、查看与导出。

## 2. 建设目标

### 2.1 业务目标
- 将原本依赖人工拆分的需求研究流程整合为统一平台。
- 通过可配置算法工具和 LLM 服务降低问卷设计、分析与报告撰写门槛。
- 保证研究过程具备可追溯、可复现、可扩展的工程特性。

### 2.2 交付目标
- 交付一套可运行的 Web 系统。
- 交付与系统现状一致的实施、技术、模块和总结文档。
- 交付可支撑阶段演示、验收和二次迭代的项目资料。

## 3. 当前系统实施范围

### 3.1 已落地范围
- 用户认证与基础权限控制。
- 项目管理、项目详情、里程碑事件展示。
- 决策管线创建、状态推进与详情页。
- 对话式需求澄清与上下文沉淀。
- 算法工具管理与选择。
- 问卷生成、编辑、发布、填写、关闭。
- 数据采集总览、题目分布统计、渠道贡献分析。
- 计算结果产物管理。
- 基于 LLM 的报告生成与项目总报生成。
- 面向 QFD 的多轮工作流与控制计划映射。

### 3.2 当前边界
- 现阶段主路径围绕 Kano、QFD、AHP 等需求/决策分析场景展开。
- 系统已支持算法工具注册，但算法生态仍以平台内置与受控扩展为主。

## 4. 总体实施路线

### 4.1 实施思路
项目采用“平台底座先行、业务链路打通、模块逐步增强”的方式推进。项目已完成从基础认证、数据模型、管线编排，到问卷、采集、报告的主链路闭环。

### 4.2 实施分层
- 表现层：Next.js App Router 页面、交互组件、报表与工作流界面。
- 应用层：Route Handlers、Server Actions、页面级流程控制。
- 领域层：Pipeline、Questionnaire、Data Acquisition、Calc、Report、Workflow 等服务。
- 基础设施层：Prisma、PostgreSQL、NextAuth、LLM Provider、外部算法工具接口。

### 4.3 关键实施节点
1. 完成项目、用户、管线等核心数据模型。
2. 完成对话澄清与上下文分析链路。
3. 完成问卷生成、编辑与发布链路。
4. 完成公开问卷填写与数据采集统计链路。
5. 完成计算结果、QFD 工作流与报告生成链路。
6. 完成系统配置、算法工具管理和项目总报能力。

## 6. 实施保障

### 6.1 技术保障
- 使用 Prisma 管理模型和迁移，保证数据结构可演进。
- 使用 NextAuth 管理身份认证，保证受限页面与接口安全。
- 使用统一服务层封装问卷、采集、报告和工作流逻辑，降低页面层耦合。

### 6.2 过程保障
- 关键业务对象均支持版本化或状态化管理。
- 采集、计算、报告阶段采用明确状态流转，避免越级操作。
- 报告和项目总报均保留结构化内容，支持再次渲染和导出。
